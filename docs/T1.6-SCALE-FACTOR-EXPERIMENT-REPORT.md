# T1.6 Scale Factor Reduction Experiment Report

**Date:** 2026-03-29
**Status:** Executed with initial empirical data

## Goal

Evaluate the tradeoff of lowering CKKS `global_scale` from `2^40` to `2^35` and `2^30`:

1. HE matmul speed
2. End-to-end accuracy tests
3. 2048x2048 matmul error growth

## What Was Added

To make this experiment executable in-repo, the following support was added:

- `client/encryption/ckks_context.py`
  Adds `load_ckks_config()` and a `global_scale_override` path for context creation.

- `benchmarks/bench_he_matmul.py`
  Adds `--config-path` so the benchmark can run against temporary CKKS configs with alternate scales.

- `client/tests/test_e2e_accuracy.py`
  Reads `ZKLLM_CKKS_SCALE` so the same test file can be exercised at different scales without copying it.

- `benchmarks/scale_experiment.py`
  New experiment runner that:
  - sweeps `2^40`, `2^35`, `2^30`
  - runs `bench_he_matmul.py` for each scale
  - runs `client/tests/test_e2e_accuracy.py` for each scale
  - measures 2048x2048 synthetic matmul error
  - writes a structured JSON report

## Empirical Results

The experiment was executed with:

- `samples=1`
- `warmups=0`
- synthetic 2048x2048 matmul error measurement
- in-process accuracy checks equivalent to the current `client/tests/test_e2e_accuracy.py` coverage areas

Key result file:

- `benchmarks/results/scale_experiment.json`

Observed 2048x2048 matmul timings:

- `2^40`: `6491.79 ms`
- `2^35`: `6479.622 ms`
- `2^30`: `6554.057 ms`

Observed relative timing change vs `2^40`:

- `2^35`: about `0.19%` faster
- `2^30`: about `0.96%` slower

Observed 2048x2048 error growth:

- `2^40`: `MAE 1.48e-09`, `max abs 1.30e-08`
- `2^35`: `MAE 3.52e-03`, `max abs 1.60e-02`
- `2^30`: `MAE 3.63e-03`, `max abs 1.65e-02`

Accuracy gate outcome in this initial run:

- All current checks passed at `2^40`, `2^35`, and `2^30`
- But the lower scales increased error by several orders of magnitude while showing no material speed gain in the measured 2048x2048 path

## How To Re-Run

```bash
python benchmarks/scale_experiment.py
```

This writes:

- `benchmarks/results/scale_experiment.json`
- per-scale matmul benchmark JSON files under `benchmarks/results/`

## Recommendation

This initial empirical run is a **no-go** for changing the default scale in Phase 1.

Recommendation:

- keep `2^40` as the baseline
- do not adopt `2^35` based on the current evidence
- do not adopt `2^30`

Reasoning:

- the measured speed difference is negligible at `2^35`
- `2^30` was slower in this run
- both lower scales materially increase approximation error

If this topic is revisited later, the next step should be a higher-sample rerun in a controlled environment, but the current data does not justify a default-scale change.
