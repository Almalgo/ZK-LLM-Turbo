# T1.4 Selective Galois Keys Finding

**Date:** 2026-03-29
**Status:** Investigated, defer to OpenFHE / lower-level SEAL access

## Summary

The Phase 1 question was whether the current TenSEAL stack supports generating only the subset of Galois keys needed for our current matrix-multiply dimensions, for example via an API like `generate_galois_keys(steps=[...])`.

For the current repository and documented TenSEAL Python API, the answer is **no clear supported path was found**.

Our code currently uses:

- [ckks_context.py](/home/rojo/Documents/Repos/ZK-LLM-Turbo/client/encryption/ckks_context.py): `context.generate_galois_keys()`

The official TenSEAL examples and docs show `generate_galois_keys()` with no step-selection argument. I did not find documented Python-level support for selective step lists in the current TenSEAL API surface we rely on.

## Evidence

Primary sources reviewed:

- TenSEAL docs: https://openmined.github.io/TenSEAL/
- TenSEAL GitHub README: https://github.com/OpenMined/TenSEAL

What those sources show:

1. The documented setup path is `context.generate_galois_keys()` with no selective-steps parameter.
2. TenSEAL exposes vector-matrix multiply and a high-level Python API, but the public docs do not document selective Galois-key generation for Python usage.
3. Our current repository uses only the documented no-argument form.

## Conclusion

For this codebase, `T1.4` should be treated as:

- **No implementation change in Phase 1**
- **Defer to T3.1 / OpenFHE or direct lower-level SEAL access**

That is the first point where we are likely to have a realistic path to explicit rotation-key selection and tighter control over evaluation-key material.

## Impact

The research note's original 50-70% estimate does not apply here anyway. At our current `poly_modulus_degree=8192`, the expected reduction is modest, and without a documented/selectable Python API in TenSEAL, the engineering path is not justified in Phase 1.

## Decision

`T1.4` is complete as an investigation task.

Result:

- keep `context.generate_galois_keys()` as-is in Phase 1,
- do not add unsupported selective-key plumbing to the client config,
- revisit this after backend migration work in Phase 3.
