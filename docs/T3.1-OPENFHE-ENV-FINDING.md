# T3.1 OpenFHE Environment Finding

Date: 2026-04-12

## Summary

The repo has a backend abstraction layer in [`common/he_backend.py`](../common/he_backend.py) and now includes a partial OpenFHE vector path, but full OpenFHE migration is still blocked.

## What was tried

- Installed `openfhe` into the project `.venv`
- Attempted `import openfhe`
- Added partial OpenFHE backend coverage for context/vector lifecycle paths in `common/he_backend.py`
- Added tests covering OpenFHE vector wrapper behavior in `client/tests/test_he_backend.py`

## Result

The package installs, but import fails immediately on this host:

```text
ModuleNotFoundError: No module named 'openfhe.openfhe'
```

This indicates the published wheel available to this environment is missing its native extension module for the current interpreter/platform combination.

The backend now also marks OpenFHE `matmul()` as intentionally unimplemented, because the OpenFHE equivalent requires explicit rotation/multiply/add evaluation loops and cannot reuse TenSEAL's `CKKSVector.mm()` API.

## Impact

- TenSEAL remains the active inference backend.
- OpenFHE smoke-level operations are available in the abstraction (`create_context`, serialize/deserialize context and vector, `encrypt`, `decrypt`, `clone`, `square`).
- A real OpenFHE inference backend still requires both a working OpenFHE build and an implemented `matmul()` path.

## Next step

To finish `T3.1`, use one of:

- a compatible Python version / OpenFHE wheel combination
- a source build of OpenFHE bindings
- a containerized environment where OpenFHE is known-good

Then implement and benchmark OpenFHE `matmul()` in `common/he_backend.py` with accuracy validation against the current TenSEAL baseline.
