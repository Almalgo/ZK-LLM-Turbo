# T3.1 OpenFHE Environment Finding

Date: 2026-04-12

## Summary

The repo has a backend abstraction layer in [`common/he_backend.py`](../common/he_backend.py) and now includes an executable OpenFHE vector path plus an initial OpenFHE matmul implementation.

OpenFHE import is now working in this environment, but the path is still experimental and not benchmark-validated for production.

Backend comparison artifacts currently show near-identical numeric error, but significantly slower runtime for OpenFHE than TenSEAL in this initial implementation. Recent larger-dimension signals:

- `2048x256`: OpenFHE `112646.797ms` vs TenSEAL `13193.391ms` (~8.5x slower)
- `2048x256` (2-run sample with OpenFHE plaintext-column cache reuse): OpenFHE `96635.507ms` vs TenSEAL `12551.124ms` (~7.7x slower)
- `2048x256` (2-run sample with plaintext cache + context key-state cache): OpenFHE `115636.227ms` vs TenSEAL `13035.419ms` (~8.9x slower)
- `2048x1024`: OpenFHE `384243.858ms` vs TenSEAL `12882.026ms` (~29.8x slower)
- `2048x2048`: OpenFHE `491582.044ms` vs TenSEAL `15208.438ms` (~32.3x slower)

## Result

`openfhe` now imports successfully on this host.

The backend smoke probe now passes for:

- context creation
- public context serialization roundtrip
- encrypt/decrypt roundtrip
- `square()`
- `matmul()` (small smoke case)

## Impact

- TenSEAL remains the active inference backend.
- OpenFHE operations are available in the abstraction (`create_context`, serialize/deserialize context and vector, `encrypt`, `decrypt`, `clone`, `square`, `matmul`).
- OpenFHE `matmul()` is currently experimental and needs larger-dimension benchmarks and accuracy gates before production use.
- Current readiness report decision is `no_go` for production default due performance (`worst_slowdown ~32.3x` vs TenSEAL).

## Next step

To finish `T3.1` with production confidence:

- benchmark OpenFHE `matmul()` at realistic dimensions (`2048x256`, `2048x2048`, FFN widths)
- run prompt-level parity checks against TenSEAL baseline across multiple layers/tokens
- document throughput and memory tradeoffs before switching default backend
