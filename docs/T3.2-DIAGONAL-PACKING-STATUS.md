# T3.2 Diagonal Packing Status

Date: 2026-04-12

## What landed

- Added cyclic diagonal extraction helpers to [`server/model/weight_manager.py`](../server/model/weight_manager.py)
- Added caching for per-layer diagonal weights via `get_layer_diagonal_weights()`
- Added unit tests for diagonal extraction and caching
- Added `he_matmul_diagonal()` runtime prototype in [`server/inference/he_ops.py`](../server/inference/he_ops.py) with correctness test coverage
- Added prototype benchmark artifact [`benchmarks/results/bench_diagonal_runtime_64.json`](../benchmarks/results/bench_diagonal_runtime_64.json)

## What is still blocked

The runtime packed-matmul path is still blocked on completion-grade backend migration from `T3.1`.

Reason:

- TenSEAL's `CKKSVector.mm()` does not expose the manual rotate / multiply / accumulate control needed for THOR-style diagonal packing.
- OpenFHE is now importable and has an initial `matmul()` path, but we still need larger-dimension parity/performance validation before using it as the base for diagonal-packing runtime changes.

## Conclusion

`T3.2` has landed its preprocessing groundwork, but not the runtime diagonal packed matmul implementation. Completing the task now depends on production-grade OpenFHE validation and then implementing the packed runtime path on top of that backend.

Current benchmark signal confirms the prototype is wiring-correct but not optimized (small-dimension slowdown vs dense path).
