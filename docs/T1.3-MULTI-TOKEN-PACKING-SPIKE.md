# T1.3 Multi-Token Packing Spike

**Date:** 2026-03-29
**Status:** Investigated, not shippable with current TenSEAL matmul path

## Summary

The Phase 1 action plan proposed packing two 2048-dim tokens into one 4096-slot CKKS ciphertext during prefill so the server could process both tokens with no HE logic changes.

That assumption does **not** hold for the current implementation.

Our server uses TenSEAL's `CKKSVector.mm()` / `matmul()` interface for all encrypted linear layers. This is a vector-matrix multiply, not an element-wise slot-local transform. Concatenating two 2048-dim tokens into a single 4096-value encrypted vector changes the encrypted vector's logical length from 2048 to 4096. The server weight matrices are shaped for 2048-row inputs, so the packed ciphertext no longer matches the current `mm()` contract.

## Evidence

1. The current HE path calls `enc_vector.mm(weight_matrix)` through [he_ops.py](/home/rojo/Documents/Repos/ZK-LLM-Turbo/server/inference/he_ops.py).
2. TenSEAL documents `matmul` as vector-matrix multiplication on the encrypted vector, not as an operation that independently applies the same matrix to disjoint slot regions.
3. The current server weights for Q/K/V/O/Gate/Up are all built for 2048 input rows; FFN down uses explicit split-input handling, but still assumes each ciphertext corresponds to one logical chunk.

Primary sources used for the spike:
- TenSEAL README / GitHub: https://github.com/OpenMined/TenSEAL
- TenSEAL docs site: https://openmined.github.io/TenSEAL/

## Consequence

The client-side concatenation packing attempted in commit `d88d95f` was incorrect and was reverted in commit `e6aa186`.

The helper functions in [layer_protocol.py](/home/rojo/Documents/Repos/ZK-LLM-Turbo/client/inference/layer_protocol.py) remain as exploratory groundwork only. They are **not** used in the live inference path.

## What Would Make T1.3 Feasible

One of the following would be required:

1. **Server-assisted unpack before matmul**
   The server would need supported TenSEAL/SEAL operations to split one packed ciphertext back into two logical 2048-dim encrypted subvectors before calling `.mm()`.

2. **Different packing layout plus matching weight encoding**
   This is closer to diagonal / SIMD-aware packing and would require matching changes in how weights are encoded and multiplied.

3. **Backend migration**
   A lower-level backend such as OpenFHE or direct SEAL APIs could support a custom packed matmul implementation.

## Decision

`T1.3` is **not complete** under the current TenSEAL Phase 1 architecture.

For this repository state, the safe conclusion is:
- keep the runtime inference path un-packed,
- keep the exploratory helper/tests as non-runtime groundwork,
- treat true multi-token packing as blocked on a different HE matmul design or backend capability.
